# Git LFS 配额超限问题说明

## 为什么会有配额超限？

### GitHub LFS 配额限制

GitHub 对 Git LFS 有严格的配额限制：

| 账户类型 | 存储空间 | 每月带宽 |
|---------|---------|---------|
| 免费账户 | 1 GB | 1 GB |
| Pro 账户 | 50 GB | 50 GB |
| Team 账户 | 50 GB | 50 GB |
| Enterprise | 根据计划 | 根据计划 |

### BODex 仓库的情况

- **LFS 文件总数**: 304 个
- **文件类型**: 主要是机器人网格文件（.obj, .stl, .ply）
- **文件大小**: 每个文件通常几 MB 到几十 MB

**估算**：
- 如果平均每个文件 2-3 MB
- 304 个文件 × 2.5 MB ≈ **760 MB**
- 这已经接近免费账户的 1 GB 限制

### 超限的原因

1. **存储空间不足**
   - 所有 LFS 文件总大小超过账户配额
   - 免费账户只有 1 GB，很容易超限

2. **带宽限制**
   - 即使存储空间够，每月下载带宽也可能超限
   - 每次 `git lfs pull` 都会消耗带宽配额
   - 如果多人使用，带宽消耗更快

3. **文件累积**
   - 随着项目更新，LFS 文件可能越来越多
   - 历史文件没有清理，占用配额

### 为什么现在无法拉取？

当配额超限时：
- GitHub 会拒绝所有新的 LFS 请求
- 显示 "exceeded its LFS budget" 错误
- 无法拉取任何 LFS 文件，即使只是部分文件

### 解决方案

1. **增加配额**（推荐）
   - 升级到 GitHub Pro（$4/月）
   - 或联系仓库维护者增加配额

2. **使用替代存储**
   - 将大文件移到外部存储（Google Drive、百度网盘等）
   - 在 README 中提供下载链接

3. **优化文件**
   - 压缩 STL/OBJ 文件
   - 删除不必要的文件
   - 使用更高效的格式

4. **等待配额重置**
   - 带宽配额每月重置
   - 但存储空间不会自动重置
